# PromptClip 桌面版多端实现方案

## 结论

当前项目具备演进为“一套代码同时构建 Web、macOS、Windows 桌面应用”的基础，推荐采用 **Tauri 2 + React/Vite 共享前端** 的方案。

需要注意的是，PRD 中“前端代码 100% 复用”的表述应调整为：

- UI、状态管理、Markdown 解析、搜索、标签等应用层逻辑可以高度复用。
- 文件系统、目录选择、权限恢复、持久化、剪贴板、导出下载等平台能力需要通过适配层隔离。

因此，这不是重写项目，而是一次以“文件访问抽象”为核心的架构改造。

## 当前项目现状

当前项目是纯 Web Vite 应用：

- 前端框架：React 18 + TypeScript
- 构建工具：Vite
- 状态管理：Zustand
- 样式：Tailwind CSS
- 搜索：FlexSearch
- 数据文件：Markdown + YAML frontmatter
- 文件访问：浏览器 File System Access API

当前构建脚本仍然只有 Web 相关命令：

```json
{
  "dev": "vite",
  "build": "tsc && vite build",
  "preview": "vite preview",
  "type-check": "tsc --noEmit",
  "lint": "eslint src --ext .ts,.tsx",
  "test": "vitest"
}
```

当前还没有 Tauri scaffold，也没有桌面端构建入口。

## 主要平台耦合点

当前代码中，大部分平台耦合集中在文件访问相关模块。

| 模块 | 当前情况 | 迁移影响 |
| --- | --- | --- |
| `src/services/fileService.ts` | 直接使用 `showDirectoryPicker`、`FileSystemDirectoryHandle`、`FileSystemFileHandle` | 桌面端无法直接复用，需要抽象 |
| `src/services/promptService.ts` | Prompt CRUD 参数和返回值依赖浏览器文件句柄 | 业务层被 Web API 污染，需要改为平台无关路径/文件记录 |
| `src/types/prompt.ts` | `Prompt` 实体包含 `fileHandle?: FileSystemFileHandle` | Domain 类型不应包含浏览器专属句柄 |
| `src/stores/fileStore.ts` | 使用 IndexedDB 保存目录句柄，localStorage 保存状态 | Web 可保留，桌面端应保存真实目录路径 |
| `src/hooks/useDirectoryPicker.ts` | 直接调用浏览器目录选择 API | 需要改为调用平台文件仓储 |
| `src/hooks/usePromptLoader.ts` | 直接从 store 获取 `directoryHandle` | 应改为获取平台无关的 workspace/root reference |
| `src/services/folderConfigService.ts` | `.promptclip.json` 读写依赖目录句柄 | 配置逻辑可复用，底层读写需要换成仓储接口 |

## 可复用部分

以下模块和能力可以高度复用：

- React 组件布局和交互
- Zustand 的 `promptStore`、`tagStore`、`uiStore` 大部分逻辑
- Markdown frontmatter 解析和序列化
- FlexSearch 搜索索引
- 标签树构建、标签筛选、收藏、复制计数等业务规则
- 回收站、历史版本等文件组织规则
- Tailwind 样式和现有 UI 组件

## 需要改造部分

需要优先改造的部分：

1. 从 `Prompt` 类型中移除 `fileHandle`
2. 从 `PromptService` 中移除对 `FileSystemDirectoryHandle`、`FileSystemFileHandle` 的直接依赖
3. 新增平台无关的文件仓储接口
4. 保留 Web 文件访问实现，新增 Tauri 桌面文件访问实现
5. 将 `fileStore` 从“浏览器目录句柄状态”调整为“当前工作区状态”
6. 将目录选择、权限校验、目录恢复做成平台适配能力

## 推荐目标架构

建议增加一个平台无关的文件仓储接口，例如：

```ts
export interface WorkspaceRef {
  id: string;
  name: string;
  path?: string;
  handle?: unknown;
  platform: 'web' | 'desktop';
}

export interface FileEntry {
  name: string;
  path: string;
  size: number;
  modifiedAt: Date;
}

export interface FileRepository {
  isSupported(): boolean;
  selectDirectory(): Promise<WorkspaceRef | null>;
  restoreDirectory(): Promise<WorkspaceRef | null>;
  verifyPermission(workspace: WorkspaceRef): Promise<boolean>;

  listFiles(workspace: WorkspaceRef, extensions: string[]): Promise<FileEntry[]>;
  readText(workspace: WorkspaceRef, path: string): Promise<string>;
  writeText(workspace: WorkspaceRef, path: string, content: string): Promise<FileEntry>;
  exists(workspace: WorkspaceRef, path: string): Promise<boolean>;
  move(workspace: WorkspaceRef, from: string, to: string): Promise<void>;
  mkdir(workspace: WorkspaceRef, path: string): Promise<void>;
}
```

然后实现两套 adapter：

| 平台 | 实现方式 |
| --- | --- |
| Web | `webFileRepository`，内部继续使用 File System Access API + IndexedDB |
| macOS / Windows | `tauriFileRepository`，通过 Tauri dialog/fs/path/store 或 Rust command 访问真实路径 |

业务层只依赖 `FileRepository`，不直接依赖浏览器或 Tauri API。

## Prompt 类型调整建议

当前：

```ts
export interface Prompt {
  id: string;
  title: string;
  content: string;
  tags: string[];
  createdAt: Date;
  updatedAt: Date;
  copyCount: number;
  pinned: boolean;
  fileHandle?: FileSystemFileHandle;
}
```

建议改为：

```ts
export interface Prompt {
  id: string;
  title: string;
  content: string;
  tags: string[];
  createdAt: Date;
  updatedAt: Date;
  copyCount: number;
  pinned: boolean;
  filePath: string;
}
```

这样 `Prompt` 成为纯业务实体，不再绑定某个平台的文件句柄。

## 为什么推荐 Tauri 2

Tauri 2 适合当前项目，原因是：

- 当前项目已经是 Vite + React，接入成本较低。
- PromptClip 的核心能力是本地文件读写，Tauri 对 macOS/Windows 原生文件访问比浏览器 FSAA 更稳定。
- 桌面端包体和资源占用比 Electron 更适合个人效率工具。
- Web 版可以继续保留当前浏览器能力，不需要为桌面端牺牲 Web 体验。
- 后续可以自然扩展系统托盘、全局快捷键、文件监听、自动更新等桌面能力。

不建议桌面端继续依赖 File System Access API。Tauri WebView 对浏览器文件 API 的支持、权限模型和持久化行为都不应作为核心依赖。

## 桌面端实现路径

### 第一阶段：文件访问抽象

目标：不引入 Tauri，先把 Web API 从业务层剥离。

主要任务：

- 新增 `FileRepository` 接口
- 新增 `WorkspaceRef`、`FileEntry` 等平台无关类型
- 将 `PromptService` 改为依赖文件路径和仓储接口
- 将 `Prompt.fileHandle` 替换为 `Prompt.filePath`
- 保持现有 Web 功能不变

这一阶段完成后，Web 版仍然可以正常运行，但业务层已经具备多端基础。

### 第二阶段：Web adapter

目标：把现有 File System Access API 封装为 `webFileRepository`。

主要任务：

- 将当前 `fileService.ts` 改造成 Web adapter
- IndexedDB 继续用于保存浏览器目录句柄
- 保留 `showDirectoryPicker` 选择目录流程
- 保留 Chrome / Edge 优先支持策略

### 第三阶段：Tauri scaffold

目标：让项目具备桌面端开发和打包能力。

主要任务：

- 初始化 Tauri 2
- 增加桌面端脚本，例如：

```json
{
  "tauri:dev": "tauri dev",
  "tauri:build": "tauri build"
}
```

- 配置 Tauri app metadata
- 配置 macOS / Windows 打包参数
- 配置文件系统权限 scope

### 第四阶段：Desktop adapter

目标：实现 macOS / Windows 原生文件读写。

主要任务：

- 使用 Tauri dialog 选择目录
- 使用 Tauri fs 或 Rust command 递归扫描 `.md` 文件
- 使用真实路径读写 Markdown 文件
- 使用 Tauri store 或配置文件保存最近工作区路径
- 桌面端启动时尝试恢复上次目录

### 第五阶段：构建验证

目标：验证三端构建和核心功能。

需要验证：

- Web：`npm run build`
- macOS：`npm run tauri:build`
- Windows：`npm run tauri:build`
- 选择目录
- 加载 Markdown prompts
- 新建、编辑、删除
- 收藏、复制计数
- 标签置顶配置 `.promptclip.json`
- 历史版本 `.history/`
- 回收站 `.trash/`

## 主要风险

### Web 兼容性

File System Access API 主要适合 Chromium 系浏览器。Safari、Firefox 仍不是可靠目标。

建议 Web 版明确定位为：

- Chrome
- Edge
- 其他支持 FSAA 的 Chromium 浏览器

### 桌面安全权限

Tauri 2 的文件访问需要配置权限 scope。应用不应默认拥有全盘读写权限。

建议只授权用户选择的数据目录，并将权限范围限制在该目录内。

### 路径差异

macOS、Windows 的路径规则不同：

- 路径分隔符不同
- Windows 有保留文件名
- 非法字符规则不同
- 大小写敏感性不同

文件名生成、路径拼接、重命名、删除等逻辑需要集中放在工具函数里处理。

### 目录持久化模型不同

Web 端保存的是 `FileSystemDirectoryHandle`，桌面端保存的是真实路径。

不能试图复用同一种持久化数据结构。应统一抽象为 `WorkspaceRef`，底层分别保存。

### 文件监听

当前项目明确“无文件监听”，用户在文件夹中手动增删文件后需要刷新页面。

桌面端虽然可以实现文件监听，但不建议第一阶段引入。应先完成多端文件读写和构建，再考虑 watcher。

## 工作量评估

如果目标是“Web + macOS + Windows 都能构建并完成核心 CRUD”，迁移量为中等，不是重写。

粗略估计：

| 工作项 | 估计时间 |
| --- | --- |
| 文件仓储抽象与 Web 适配 | 1-2 天 |
| Tauri scaffold 与桌面 FS 适配 | 2-4 天 |
| macOS / Windows 构建与权限调试 | 1-3 天 |
| 稳定性测试和路径边界修正 | 2-4 天 |

整体预计：约 1-2 周完成可用版本。

## 推荐实施顺序

1. 先做平台无关文件仓储接口，把 `FileSystemDirectoryHandle` / `FileSystemFileHandle` 从 `PromptService` 和 `Prompt` 类型中移除。
2. 保留现有 Web 行为，实现 `webFileRepository`，确保 Web 功能不回退。
3. 接入 Tauri 2 scaffold，新增桌面开发和构建脚本。
4. 实现 `tauriFileRepository`，桌面端使用真实路径读写 Markdown。
5. 做 macOS / Windows 打包验证。
6. 后续再考虑代码签名、自动更新、全局快捷键、系统托盘、文件监听、SQLite 索引缓存。

## 最终判断

当前代码基础适合演进到一套代码多端构建。

最关键的第一步不是直接接入 Tauri，而是先把浏览器文件句柄从业务模型和服务层中剥离。只要完成这一步，Web、macOS、Windows 的差异就可以被限制在 adapter 层，后续桌面化会变得清晰、可控。
